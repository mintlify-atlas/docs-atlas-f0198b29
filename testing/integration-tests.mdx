## Complete Voting Cycle Test

The primary integration test validates a full election cycle with reset:

```solidity
function testIfVoterCanVoteAfterReset() external {
    (uint256 entryFee) = helperConfig.activeNetworkConfig();
    
    // First election cycle
    vm.prank(msg.sender);
    memberVote.startVote();
    vm.prank(USER1);
    memberVote.vote{value: entryFee}(OPTION_A);
    vm.prank(USER2);
    memberVote.vote{value: entryFee}(OPTION_B);
    vm.prank(USER3);
    memberVote.vote{value: entryFee}(OPTION_A);

    // Reset and start new election
    vm.startPrank(msg.sender);
    memberVote.resetVotes();
    memberVote.startVote();
    vm.stopPrank();
    
    // Second election cycle
    vm.prank(USER1);
    memberVote.vote{value: entryFee}(OPTION_A);
    vm.prank(USER2);
    memberVote.vote{value: entryFee}(OPTION_B);
    vm.prank(USER3);
    memberVote.vote{value: entryFee}(OPTION_A);
    
    assertEq(memberVote.getOptionAVotes(), 2, "Option A should have 2 votes after reset and new votes");
    assertEq(memberVote.getOptionBVotes(), 1, "Option B should have 1 vote after reset and new votes");
}
```

## What This Test Validates

### Complete Workflow

The test validates the entire voting workflow:

1. **Initial Election**
   - Owner starts voting session
   - Multiple users cast votes
   - Votes are counted correctly
   - First election has results: 2 votes for Option A, 1 for Option B

2. **Reset Process**
   - Owner resets votes (transitions to Finished state)
   - Owner starts new voting session
   - State properly transitions back to Voting

3. **New Election**
   - Same users can vote again in new election
   - Previous votes don't carry over
   - New votes are counted independently
   - Second election has fresh results

### State Reset Verification

The test confirms that `resetVotes()` properly clears:

- Vote counts (both options reset to 0)
- Voter tracking (users can vote again)
- Election ID (increments to track new election)
- Workflow station (transitions through states correctly)

### Multi-User Scenarios

The test uses three different users to validate:

- Multiple voters can participate in same election
- Each user's vote is counted correctly
- Users can participate in multiple elections
- Duplicate vote prevention works within each election but not across elections

### State Transition Flow

The complete state flow validated:

```
Registering (0) 
    → startVote() → 
Voting (1) 
    → [users vote] → 
    → resetVotes() → 
Finished (2) 
    → startVote() → 
Voting (1) 
    → [users vote again]
```

## Key Differences from Unit Tests

### Unit Tests
- Test individual functions in isolation
- Validate single operations
- Focus on edge cases and error conditions
- Quick to run
- Easy to pinpoint failures

### Integration Tests
- Test complete workflows
- Validate multi-step processes
- Focus on real-world scenarios
- May take longer to run
- Test interactions between components

## Best Practices

### Descriptive Assertions

Always include meaningful error messages:

```solidity
assertEq(
    memberVote.getOptionAVotes(), 
    2, 
    "Option A should have 2 votes after reset and new votes"
);
```

This helps identify which part of a complex workflow failed.

### Multiple Verification Points

Integration tests should verify state at multiple points:

```solidity
// After first election (implicit in the test flow)
// After reset (could add explicit checks)
// After second election (explicitly verified)
assertEq(memberVote.getOptionAVotes(), 2);
assertEq(memberVote.getOptionBVotes(), 1);
```

### Use vm.startPrank/stopPrank for Multiple Calls

When the same address makes multiple calls:

```solidity
vm.startPrank(msg.sender);
memberVote.resetVotes();
memberVote.startVote();
vm.stopPrank();
```

This is more efficient than multiple `vm.prank()` calls.
